# Implementation Report - March 21, 2026

## Scope
This report summarizes the implementation completed across the Builder preview runtime and the AI requirements flow, including hardening, bug fixes, behavioral changes, and verification outcomes.

---

## 1) Preview Pipeline Migration and Hardening

### 1.1 Sandpack Removal and Local Preview Runtime
- Replaced Sandpack-based preview rendering path with an in-house iframe runtime at `/builder/preview-runner`.
- Parent orchestrator sends code/files through `postMessage`; iframe compiles and renders locally.
- Removed dependency on external CodeSandbox execution path for preview runtime reliability.

Primary files:
- `frontend/pages/builder/ai-generate.tsx`
- `frontend/pages/builder/preview-runner.tsx`
- `frontend/lib/previewUtils.ts`
- `OWN_PREVIEW_PIPELINE.md`

### 1.2 Stack-Aware Rendering
- Added stack detection and stack-aware rendering behavior for:
  - React/Next-like output
  - Vue output
  - HTML output
- Added stack signal messaging to parent using `PREVIEW_MSG.STACK_DETECTED`.
- Added stack diagnostics in parent preview header.

Key markers:
- `frontend/pages/builder/preview-runner.tsx:1333` (`STACK_DETECTED` postMessage)
- `frontend/pages/builder/preview-runner.tsx:7` (`PREVIEW_MSG` usage)

### 1.3 Runtime Stability and Cache Correctness
- Added module cache clear at start of each preview update to avoid stale module reuse across regenerate/refine cycles.
- Added ping/pong handler support for parent-child heartbeat messaging.

Key markers:
- `frontend/pages/builder/preview-runner.tsx:1288` (`moduleCache.clear()`)
- `frontend/pages/builder/preview-runner.tsx` message listener includes PING -> PONG behavior

### 1.4 Auth Context and React Runtime Crash Fixes
- Fixed `_context` runtime errors by hardening context handling in preview runtime.
- Added robust AuthContext request detection and normalization for different import path variants.
- Added safe fallback for invalid `useContext(...)` inputs in sandbox runtime.

Key markers:
- `frontend/pages/builder/preview-runner.tsx:108` (`ensureSafeUseContext`)
- `frontend/pages/builder/preview-runner.tsx:727` (`isAuthContextRequest`)
- `frontend/pages/builder/preview-runner.tsx:853` (AuthContext request interception)

### 1.5 React Root Reuse and CreateRoot Warning Fix
- Eliminated duplicate `createRoot()` warnings by reusing a single root instance and attaching cached root handle to mount node.

Key markers:
- `frontend/pages/builder/preview-runner.tsx:410`
- `frontend/pages/builder/preview-runner.tsx:1129`

### 1.6 Constructor-Safe Module Mocks
- Fixed "Class extends value ... is not a constructor or null" by replacing arrow-function mock exports with constructable function component mocks.
- Applied to generic unresolved module mock and MUI component mock path.

Primary file:
- `frontend/pages/builder/preview-runner.tsx`

## 2) AI Requirements Flow Improvements

### 2.1 Project Name Sanitization and Display Cleanup
- Added project slug sanitization so conversational phrases do not become project names.
- Strips prefixes such as:
  - "I want to build..."
  - "want build..."
  - "create a/the..."
  - slug variants (`want-build-*`, `i-want-build-*`, etc.)
- Applied both at data-source level and display formatting level.

Key markers:
- `frontend/hooks/useRequirementsFlow.ts:40` (`sanitizeProjectSlug`)
- `frontend/hooks/useRequirementsFlow.ts:118` (sanitized projectName assignment)
- `frontend/pages/builder/select-ai.tsx:39` (`stripConversationalPrefixes`)
- `frontend/pages/builder/select-ai.tsx:345` (rendered heading)

### 2.2 AI-Only Question Generation Policy
- Removed deterministic hardcoded fallback question generation for `/api/ai/requirements`.
- Questions are now generated by AI only.
- On provider limit/malformed AI output, endpoint returns explicit errors instead of synthetic question sets.

Primary file:
- `backend/src/modules/ai/ai.service.ts`

### 2.3 Better Provider-Limit Handling for Requirements Questions
- Added non-streaming Gemini fallback model chain retry behavior to improve success rate under quota/high-demand conditions.
- Updated controller to return `429` for provider-limit conditions with `Retry-After` header instead of generic `500`.

Key markers:
- `backend/src/modules/ai/ai.service.ts:615` (`generateNonStreaming`)
- `backend/src/modules/ai/ai.controller.ts:642` (`Retry-After`)
- `backend/src/modules/ai/ai.controller.ts:645` (429/500 selection)

### 2.4 End-to-End Answer Preservation in Compile Flow
- Verified user answers are sent from frontend compile payload to backend prompt.
- Enforced final output to always preserve exact user-submitted answers even if AI output is partial/rewritten or JSON repair fallback is used.

Key markers:
- `frontend/hooks/useRequirementsFlow.ts:148` (answers array creation)
- `frontend/hooks/useRequirementsFlow.ts:158` (answers payload send)
- `backend/src/modules/ai/ai.prompts.ts:702` (`answersText` prompt injection)
- `backend/src/modules/ai/ai.service.ts:802` (`parsed.answers = params.answers`)

## 5) Behavioral Outcomes

### Preview Behavior
- Local preview is resilient to many unresolved imports via runtime mocking.
- Context-related crashes and root recreation warnings were hardened.
- Stack detection and diagnostics are available for better observability.

### Requirements Behavior
- Questions are AI-generated (no deterministic fallback question set).
- Provider exhaustion now surfaces as explicit retryable API status (`429`) instead of opaque `500`.
- Compiled requirements preserve exactly what users answered.

---

## 6) Current Known Constraints

- AI-only question generation means provider availability directly impacts Step 3 question generation.
- If platform/provider is exhausted, user may need to retry, switch model/provider, or supply BYOK key.
- Preview runtime remains a client-side approximation of framework behavior and not a full production runtime.

---
